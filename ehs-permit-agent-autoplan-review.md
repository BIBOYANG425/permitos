# EHS Permit-Navigator - Product and Build Plan

Updated: 2026-05-30
Home repo: https://github.com/a1gmm/Autoresearch-Systems-Hackathon-Antler
Audience: hackathon teammates
Status: team-shareable build artifact

## One-Line Product

An AI-native EHS research swarm that takes a facility change, determines which environmental permits and compliance obligations may apply, fetches current primary regulatory sources, verifies each determination with citations and threshold math, and produces a defensible applicability matrix for human review.

## Executive Decision

Build this, but keep the hackathon demo narrow and strong.

The winning demo is not "any project, all compliance." The winning demo is:

> Given a Southern California manufacturing change, the agent swarm determines applicable EHS obligations, proves each row with current source evidence, and visibly fails closed when the evidence is incomplete.

This is strong because it hits the hackathon theme directly:

- Agent architecture: orchestrator, specialist researchers, verifier, repair loop, memory writer.
- Retrieval and synthesis: primary source fetch, quote grounding, source hashes, structured extraction.
- Applied autonomous research: environmental compliance applicability, a high-cost legal/regulatory workflow.

## Core Wedge

Applicability determination is the wedge.

The customer pain is the question:

> "Given this facility, location, equipment, chemicals, throughput, and operational change, what permits, plans, thresholds, reports, or exemptions apply?"

This is where consultants spend senior time, where in-house EHS managers miss things, and where an autonomous research system can show real value.

## Seeded Regulatory Programs

Seed 6-8 high-signal obligations instead of 30-40 shallow rows.

Recommended seed:

1. SCAQMD Rule 201 Permit to Construct/Operate.
2. SCAQMD Rule 219 exemption/exclusion check.
3. SCAQMD Rule 222 registration check.
4. California Industrial General Permit via SIC/NAICS.
5. HMBP/CERS hazardous materials thresholds.
6. Hazardous waste generator status.
7. Construction stormwater if disturbance acreage is present.
8. Wastewater pretreatment if relevant to the scenario.

Why: each row needs real research, source retrieval, extraction, predicate evaluation, and verification. Eight deep rows beat forty shallow rows.

## Recommended System Shape

Use an artifact-driven loop, not a free-form chat loop.

```
Customer Intake
  -> ScopePack
  -> CoverageFamilyStatus[]
  -> RegulatoryAngle[]
  -> ResearchHypothesis[]
  -> ResearchTasks
  -> EvidenceBundles
  -> VerificationVerdicts
  -> RepairTickets if needed
  -> ApplicabilityMatrix
  -> Report
  -> GatedMemoryWrites
```

The coverage floor is not a fixed checklist and not the hypothesis list. It is only a completeness guard that forces the planner to inspect each broad family and explain one of four outcomes:

- active: project facts justify deeper research,
- blocked_missing_fact: a required fact is missing,
- out_of_scope: the family was inspected and does not apply to this scenario,
- discovery_candidate: the family may contain an unseeded issue that requires human review.

The planner should then expand active or discovery families into regulatory angles and specific hypotheses:

```
CoverageFamily
  -> RegulatoryAngle
    -> ResearchHypothesis
      -> ResearchTask
        -> EvidenceBundle
```

Example:

```
air
  -> new emitting equipment
    -> Does the coating booth require an SCAQMD Permit to Construct?
    -> Is Rule 219 exemption available?
    -> Does Rule 222 registration apply instead?

hazmat
  -> hazardous material inventory
    -> Does 60 gallons of flammable solvent exceed HMBP thresholds?
    -> Does battery storage create CUPA disclosure or fire-code review issues?

waste
  -> spent solvent and battery handling
    -> Does spent solvent affect generator status?
    -> Is missing monthly generation a blocker?
```

High-level architecture:

```
                         +-------------------------+
                         | Next.js intake + report |
                         +-----------+-------------+
                                     |
                                     v
                         +-------------------------+
                         | Orchestrator / Planner  |
                         | scope, coverage, angles |

                +--------------------+--------------------+
                |                    |                    |
                v                    v                    v
       +----------------+    +----------------+    +----------------+
       | Source task    |    | Source task    |    | Source task    |
       | for hypothesis |    | for hypothesis |    | for hypothesis |

```

## Agent Roles

These are agent templates, not a fixed headcount. The orchestrator should spawn as many scoped workers as the project needs, then enforce budgets, dedupe, and verification.

| Agent Template | Job | Must Not Do |
|---|---|---|
| Scope Agent | Convert intake into normalized facts, assumptions, and missing fields. | Decide final applicability. |
| Orchestrator | Inspect coverage families, create regulatory angles and hypotheses, assign tasks, enforce budgets, trigger repair. | Treat its own judgment as evidence. |
| Research Worker | Fetch sources, extract quotes, propose evidence-backed claims for one hypothesis or subtask. | Write final report or durable memory. |
| Verification Agent | Check source currency, authority, grounding, and predicate math. | Accept claims without quote support. |
| Synthesis Agent | Produce matrix and report from verified or flagged verdicts. | Promote unverified claims to facts. |
| Memory Writer | Store verified facts, source hashes, failed hypotheses, and run metrics. | Store unverified law as trusted knowledge. |

Research fan-out rule:

```
agent_count = specific research hypotheses + required source-check subtasks
minimum = coverage families inspected and represented in the graph
maximum = constrained by runtime, source, and model-call budgets
```

For a small renovation, this might be 4-6 research workers. For a complex EV battery recycler, it might be 25+ workers. The architecture should not hardcode a fixed agent count.

## Verification Standard

Each determination must pass these checks:

| Check | Question |
|---|---|
| Currency | Was the source fetched or confirmed current for this run? |
| Authority | Is the source official or high-authority enough for the row? |
| Grounding | Does the exact quote support the extracted claim? |
| Predicate math | Do the customer facts actually meet or miss the threshold? |
| Cross-source | For high-risk rows, does another trusted source agree? |

Rows that fail should not disappear. They should become `Needs-review` with a reason.

Example:

```
Needs review: source found, but quote did not support the extracted threshold.
Needs review: missing NAICS/SIC blocks industrial stormwater determination.
Needs review: only a secondary source was found.
```

## Demo Flow

### Scene 1: Seeded Facility Change

Presenter line:

> A metal finishing manufacturer in Los Angeles is adding equipment and storing a new chemical. The EHS manager needs to know what obligations changed before they operate.

Show:

1. Click "Load sample facility."
2. Orchestrator creates `ScopePack`.
3. Coverage floor inspects air, stormwater, hazmat, waste, wastewater.
4. Relevant families expand into regulatory angles and specific hypotheses.
5. Research agents fan out based on hypothesis-level source tasks.
6. Verifier passes some rows and flags others.
7. Applicability matrix appears.
8. Open evidence drawer for one row.
9. Show source URL, quote, hash, verifier checks, and confidence.

### Scene 2: Novel Or Incomplete Case

Presenter line:

> Now we give it an incomplete or novel project. A safe compliance agent should not guess.

Show:

1. Missing NAICS/SIC or novel process.
2. Agent creates hypothesis or Discovery candidate.
3. Verifier cannot prove it.
4. Matrix preserves the row as `Needs-review`.

Closing line:

> We do not replace the licensed consultant or EHS manager. We give them a research swarm that finds the law, proves the threshold, shows its work, and fails closed.

## Key Risks And Fixes

| Risk | Why It Matters | Fix |
|---|---|---|
| Demo looks like a rules engine. | Hackathon theme is autonomous research swarms. | Show the trace panel and evidence artifacts. |
| Agents hallucinate law. | Compliance users need defensibility. | Quote grounding and verifier gates. |
| Research loops forever. | Kills demo reliability. | Max 2 attempts per hypothesis, then `Needs-review`. |
| Memory poisons future runs. | Wrong source facts become durable. | Memory writer only stores verified, provenance-bound facts. |
| Scope sprawls. | 24-48 hours is short. | Seed fewer programs, make each row strong. |
| Legal overclaim. | Trust and liability issue. | Human-in-the-loop copy everywhere. |

## Evaluation Targets

- Applicability precision: >= 90% on seeded cases.
- Applicability recall: >= 80% on seeded cases.
- Citation grounding: >= 95% of verified rows have quote support.
- Fail-closed behavior: 100% of missing/failed evidence becomes `Needs-review`.
- Demo latency: seeded run completes in under 90 seconds.

## Source Strategy

Prioritize official sources:

- eCFR API and official federal pages.
- SCAQMD rulebook pages/PDFs.
- California Water Boards industrial stormwater pages and orders.
- CalEPA/CERS/CUPA pages.
- Local agency pages for the seeded jurisdiction.

Use mirrors or secondary sources only for cross-checking or fallback. If only a secondary source is available, mark the row `Needs-review`.
